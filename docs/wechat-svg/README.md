# 微信公众号 SVG 交互设计与排版规范

> 本目录源自 `wechat-svg-design.skill`，作为 MBEditor 项目的公众号兼容性知识库。
> 运行时对应的静态校验器见 `backend/app/services/svg_validator.py` 和 `scripts/validate_wechat_svg.py`。

这是一份**工程约束层**,不是创作模板。它告诉你在微信生态里什么**一定会失效**、什么**一定要这样写**,但具体怎么排、怎么讲故事、怎么视觉表达,由你基于项目需要自由决定。

## 核心心智模型

做微信公众号的 SVG 交互,你面对的不是标准 W3C SVG,而是**微信平台化技术体系**——W3C 规范的一个严格子集。违反子集边界的代码在编辑器保存时会被静默剥离,线上看到的是"效果没出来"而不是报错。因此:

**先守边界,再做创意**。在动笔前确认三件事:
1. 要用的动画属性在 20 个白名单内吗?
2. 要用的 HTML/CSS 特性不在禁用列表里吗?
3. 如果是跨平台发布(微博、今日头条),有没有平台差异?

不确定时,查 `references/whitelist.md` 和 `references/html-css-restrictions.md`。这两份是技术红线。

## 五大核心交互模式

任何复杂的微信 SVG 交互,拆解到底层都是这五个原型的组合。用户提需求时先想清楚属于哪一类(可以多个叠加),再决定实现路径:

| 模式 | 用途 | 何时用 |
|---|---|---|
| **伸长动画** | 点击展开/收起内容块 | 产品功能详情、FAQ、隐藏彩蛋 |
| **穿透触发** | 图层间响应关系控制 | 多层叠加交互、复杂点击热区 |
| **双层触发** | 利用 touchstart/click 的 300ms 时间差实现多步交互 | 确认型 CTA、无限选择器、选项切换 |
| **零高结构/容器** | height:0 + overflow:visible 实现无限堆叠 | 几乎所有定制级 SVG 的底层骨架 |
| **白名单动画** | 20 个 AttributeName 的原生 SVG 动画 | 基础动效、装饰、入场效果 |

每个模式的具体代码样板和注意事项在 `references/five-patterns.md`。**需要实现交互时必读该文件**,不要凭记忆写——细节(比如 `pointer-events` 的继承规则、`touchstart` 的 300ms 窗口)写错就是静默失效。

## 标准工作流

收到公众号 SVG 创作需求时,按这个顺序推进:

### 步骤 1:需求拆解

先把用户的模糊需求翻译成结构化信息:
- **内容类型**:产品介绍 / 品牌活动 / 知识科普 / 长图文 / 营销转化
- **核心 CTA**:这篇文章希望读者做什么?(下载 / 关注 / 转发 / 留资 / 阅读原文)
- **交互点**:哪些地方需要 SVG 交互?对应上面哪个模式?
- **品牌约束**:有没有现成的视觉规范、色彩、字体偏好?

如果用户信息不全,直接问。别自己脑补——微信排版的品牌一致性要求很高,猜错了要返工。

### 步骤 2:结构规划

按文章骨架搭建,**不要一上来就写动画**。推荐骨架(可根据内容类型调整):

```
1. 开篇钩子(图/短句,3秒内抓住注意力)
2. 价值主张(1-2句把"为什么要看"讲清楚)
3. 核心内容(2-4 个模块,每个模块独立成段)
4. 交互亮点(SVG 集中在这里,不要全文撒)
5. CTA 卡片(下载/关注/阅读原文)
6. 落款/品牌露出
```

**SVG 交互不是越多越好**。一篇文章 1-3 个精心设计的交互点,远好于通篇动效。多了反而劝退。

### 步骤 3:文案撰写

先把文案完整写完,再考虑交互。文案风格参考 `references/copywriting-style.md`——里面有公众号文案的常见陷阱(比如过度使用感叹号、AI 味浓的三段式、用词过于书面)和推荐写法。

### 步骤 4:SVG 代码实现

按模式对照 `references/five-patterns.md` 的样板改写。关键规则:

- **所有动画属性都必须在白名单 20 个之内**。不在白名单的属性在微信编辑器保存时会被移除。
- **禁用列表**里的 HTML/CSS 特性(`clip-path`、`mask`、`filter`、`position`、`margin` 大数值、`id` 传参等)坚决不用,用替代方案。
- **父子节点声明 xmlns**:`xmlns="http://www.w3.org/1999/xhtml"`,确保 iOS/Android 双端兼容。
- **深色模式下 SVG 模块会失效**,需要 CSS 变量手动适配,或在文案层面提示"请切换到浅色模式查看"。

### 步骤 5:兼容性自检

提交前对照 `references/pre-publish-checklist.md` 逐项检查。这是防止"本地看着好,发出来就坏"的最后一道闸。

### 步骤 6:交付

交付物通常是一个 HTML 文件,可直接粘到微信图文编辑器或 iSVG 等工具中。如果有复杂交互,建议:
- 提供可预览的 HTML(浏览器打开能看到效果)
- 附一段"使用说明":哪些地方需要在微信编辑器里二次检查、哪些素材需要替换

## 创意自由度的边界

这份 skill 规范的是**技术可行性和平台合规**,不限制:

- **视觉风格**:品牌色、字体、插画风格、版式节奏——完全自由
- **叙事结构**:开门见山 / 悬念铺垫 / 对比反差 / 场景代入——按内容最优解来
- **交互创意**:只要用白名单内的技术,怎么组合随你
- **排版节奏**:图文比例、段落长度、emoji 使用——根据受众调整

**什么情况下要守严**:技术红线(白名单、禁用清单、xmlns 声明)和平台合规(不用对抗性技术、不做留言区位移等破坏用户体验的骚操作)。

**什么情况下可以放飞**:只要不越线,视觉和创意层面大胆探索。微信 SVG 行业的标杆案例都是在规范内玩出花样的。

## 跨平台适配

如果内容要同时发布到微博头条文章、今日头条等,阅读 `references/multi-platform.md`——微博已全面开放 SVG 属性限制(仅禁用 JS),规则和微信差异很大,不能直接搬运。

## 自动校验

写完 HTML 后,跑一遍校验脚本做快速体检:

```bash
python scripts/validate_wechat_svg.py path/to/your-article.html
```

脚本会输出违规项(必修)、警告项(需人工确认)、统计信息。退出码 0 表示无违规、1 表示有违规,可以集成到交付流程。

**这个脚本不是万能的**——它只能查静态规则(属性名、禁用标签、事件处理器等),查不出"动画触发关系写错"、"交互体验不好"、"文案风格差"这类需要理解的问题。脚本通过 ≠ 可以发布。真正的发布前检查还是要对照 `references/pre-publish-checklist.md` 逐项过,并在微信编辑器里真实预览。

## 一条原则收尾

如果你发现"某个效果很酷但要用到禁用特性",**不要用对抗性技术去绕**(比如嵌 iframe、硬塞 JS)。微信会封。正确做法是:回到五大模式,用白名单内的技术重新设计实现路径。行业里所有漂亮的 SVG 作品都是这么做出来的。
